# Server- und Client-Komponenten in Next.js

## Überblick

In der letzten Session haben wir gesehen, dass Next.js Komponenten standardmäßig auf dem Server rendert. Das reicht aus, um Daten zu lesen und anzuzeigen – ohne Benutzerinteraktion fühlt sich eine App damit jedoch nicht vollständig an. Dafür brauchen wir Code, der im Browser ausgeführt wird (z. B. um auf Klicks zu reagieren oder lokalen State zu halten), Code, der auf dem Server läuft, um Änderungen in eine Datenbank zu schreiben, sowie eine Möglichkeit, Informationen zwischen beiden Seiten auszutauschen.

Eine klassische Webanwendung trennt üblicherweise ein Frontend-Projekt, das im Browser läuft, ein Backend-Projekt, das auf einem Server ausgeführt wird, und ein Netzwerk dazwischen, das du von Hand mit `fetch`-Aufrufen und API-Endpunkten überbrückst. Statt die Codebasis in eine Frontend- und eine Backend-Hälfte aufzuteilen, verwaltet Next.js alles innerhalb eines einzigen Komponentenbaums. Die meisten Komponenten laufen auf dem Server. Diejenigen, die den Browser benötigen, melden sich explizit als Client-Komponenten an. Ein Großteil der Verbindungslogik, die du normalerweise selbst schreiben würdest, um das Netzwerk zu überqueren, wird für dich generiert.

Diese Session beleuchtet diese Grenze von beiden Seiten:

1. Was die Grenze ist und wie Next.js es einer Server-Komponente ermöglicht, eine Client-Komponente wie einen normalen Import einzubinden.
2. Client-Komponenten selbst und die Regel, welche Props diese Grenze passieren dürfen.
3. Die zwei Wege, auf denen Code als Reaktion auf eine Benutzeraktion zum Server zurückreist: **Server Functions**, mit denen du Server-Code wie eine lokale Funktion aufrufen kannst, und **API Route Handler**, die dir einen einfachen HTTP-Endpunkt geben.
4. Den Abschluss bildet der Austausch des In-Memory-Mock-Services aus der letzten Session durch eine echte PostgreSQL-Datenbank, die direkt mit dem `postgres`-Client abgefragt wird.
